+# Orchestrator Evaluation — Enhancement Plan
+
+**Status:** Draft for review
+**Branch:** `feature/evaluator-confidence-tests`
+**Scope:** Improve the orchestrator's reason-between-steps capability so the automated workflow produces trustworthy, valid artifacts end-to-end.
+
+This plan addresses **one gap at a time**. Each gap is independently shippable (its own commit) and ordered by dependency: correctness first, then resilience, then output quality, then depth.
+
+---
+
+## Gap 1 — Duplicate skill narrative + single-artifact evaluation (correctness)
+
+### Problem
+1. The skill narrative is injected **twice** into the dispatch prompt:
+   - First in `invoke_skill` — `skill_invoker.py:93-100`
+   - Again inside `_build_skill_prompt` — `skill_invoker.py:179-180`
+   Wastes tokens and can confuse the model with repeated instructions.
+2. Only the **first** required artifact is evaluated — `step_executor.py:248` uses `required_artifacts[0]`. Steps that emit multiple artifacts (e.g. `code-review` → `review-spec.md`, `review-quality.md`, `human-verdict.md`) leave 2 of 3 unchecked.
+
+### Fix
+- **Single source of narrative.** Remove the narrative block from `invoke_skill` and keep it only in `_build_skill_prompt` (which already structures Context + Iron Law + Narrative + Instructions). Revert `_build_skill_prompt`'s signature back to its 4-arg form used consistently.
+- **Evaluate all artifacts.** Replace the `required_artifacts[0]` logic with a loop over every required artifact. Aggregate into a single step verdict:
+  - Step confidence = **min** of per-artifact confidences (weakest link).
+  - Step issues = union of all per-artifact issues (tagged with artifact name).
+  - Step is auto-approvable only if **every** artifact is auto-approvable.
+
+### Acceptance Criteria
+- Dispatch prompt contains the narrative exactly once (assert in a unit test).
+- A step with 3 required artifacts runs 3 evaluations; a placeholder in any one fails the step.
+- Existing `test_evaluator.py` scenarios still pass.
+
+### Risk / Effort
+Low risk, ~1 hour. Pure refactor + loop.
+
+---
+
+## Gap 2 — Retry with feedback instead of hard abort (resilience)
+
+### Problem
+The "Retry skill" path is advertised in the gate prompt but never implemented — the code comment literally says *"Would retry here (for now, proceed)"* (`step_executor.py:274-275`). Non-interactive mode simply aborts on the first failure, so a single recoverable issue (e.g. a stray `TODO`) kills the whole run.
+
+### Fix
+Introduce a **bounded retry loop** around skill dispatch:
+1. On evaluation failure, build a **feedback prompt** that appends the specific issues to the original skill prompt, e.g.:
+   > Your previous output failed validation for these reasons: [issues]. Regenerate the artifact(s) fixing ONLY these problems. Do not introduce placeholders (PLACEHOLDER/TODO/TBD).
+2. Re-dispatch the same skill with the feedback prompt.
+3. Re-evaluate. Repeat up to `MAX_RETRIES` (default **2**).
+4. If still failing after retries:
+   - **Interactive:** surface the 1/2/3 gate (proceed / retry again / abort).
+   - **Non-interactive:** abort with a clear `ESCALATE` reason recorded to the session audit.
+
+### Design Notes
+- Add `max_retries: int = 2` to `StepExecutor.__init__` (overridable via env `DEVIN_MAX_RETRIES`).
+- Track attempts per step in the session audit for empirical analysis.
+- Keep retries idempotent: each retry overwrites the artifact, not appends.
+
+### Acceptance Criteria
+- A skill that emits a placeholder on attempt 1 but clean output on attempt 2 results in a **passing** step.
+- A skill that never recovers aborts after exactly `MAX_RETRIES+1` attempts with an `ESCALATE` audit entry.
+- Retry count and final verdict are written to `session-audit.md`.
+
+### Risk / Effort
+Medium risk (loop control, infinite-loop guard), ~2-3 hours.
+

+## Gap 4 — Semantic evaluation layer (depth)
+
+### Problem
+Current checks are **heuristic** (placeholder regex, YAML/JSON parse, test-count parse). They cannot tell whether an artifact actually *satisfies the requirement*. A `design.md` with no placeholders and valid markdown scores 1.0 even if it ignores the acceptance criteria in `requirement.md`.
+
+### Fix
+Add a **semantic check** as a 5th evaluation dimension, gated behind a flag (`enable_semantic: bool`, default off until proven):
+1. Load the relevant context artifact (e.g. `requirement.md` for `writing-plans`, `design.md` for `subagent-driven-development`).
+2. Dispatch a **focused evaluation prompt** to devin (`--print`, cheap/fast model) asking a structured question:
+   > Given REQUIREMENT and ARTIFACT, does the artifact address every acceptance criterion? Respond with JSON: `{score: 0-1, missing: [...], notes: "..."}`.
+3. Parse the JSON; fold `score` into the confidence calculation (weighted, e.g. -0.3 if score < 0.6) and add `missing` items to issues.
+4. Cache the result per artifact hash to avoid repeat cost.
+
+### Design Notes
+- Semantic check is **advisory-weighted**, not absolute — keep objective checks authoritative so a model hiccup can't pass a placeholder.
+- Map each skill → which context artifact it should be evaluated against (config table).
+- Robust JSON parsing with fallback: if the eval model returns malformed output, treat as "requires_user_input" rather than crashing.
+
+### Acceptance Criteria
+- A `design.md` that omits an acceptance criterion is flagged with the specific `missing` item.
+- A compliant `design.md` passes with semantic score ≥ 0.8.
+- Semantic failures degrade confidence but never *raise* it above objective-check ceilings.
+- Disabling the flag fully reverts to current behavior (regression-safe).
+
+### Risk / Effort
+Higher risk (LLM nondeterminism, cost, parsing). ~4-5 hours + empirical tuning.
+
